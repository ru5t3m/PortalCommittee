# Техническое задание

## 1. Расшифровка сокращений и терминов

API (Application Programming Interface) - программный интерфейс приложения, через который frontend обменивается данными с backend.

Backend - серверная часть системы, отвечающая за обработку запросов, бизнес-логику, работу с базой данных, авторизацию и административные операции.

Frontend - клиентская часть системы, отображаемая пользователю в браузере.

UI (User Interface) - пользовательский интерфейс: страницы, формы, кнопки, меню, таблицы, карточки и другие визуальные элементы.

UX (User Experience) - пользовательский опыт: удобство прохождения сценариев, понятность навигации, предсказуемость действий.

JWT (JSON Web Token) - токен авторизации в формате JSON, используемый для подтверждения личности пользователя при обращении к API.

Access token - краткоживущий JWT-токен доступа, используемый frontend для авторизованных API-запросов.

Refresh token - токен обновления сессии, используемый для выпуска нового access token без повторного входа пользователя.

HttpOnly cookie - cookie, недоступная JavaScript-коду браузера; используется для более защищенного хранения refresh token.

RBAC (Role-Based Access Control) - ролевая модель доступа, при которой возможности пользователя зависят от назначенной роли.

CRUD (Create, Read, Update, Delete) - базовый набор операций над данными: создание, чтение, обновление, удаление.

RBAC-роль - роль пользователя в системе: администратор, модератор или кандидат.

БД - база данных.

SQL (Structured Query Language) - язык запросов к реляционным базам данных.

ORM (Object-Relational Mapping) - технология работы с базой данных через программные модели.

SQLAlchemy - ORM-библиотека Python, используемая backend для работы с базой данных.

Alembic - инструмент миграций базы данных, используемый для управляемого изменения структуры БД.

FastAPI - backend-фреймворк Python для разработки API.

Pydantic - библиотека Python для описания и проверки входных и выходных данных API.

Next.js - frontend-фреймворк React для построения веб-приложений.

React - библиотека JavaScript/TypeScript для построения пользовательских интерфейсов.

TypeScript - язык программирования на основе JavaScript со статической типизацией.

Tailwind CSS - CSS-фреймворк для стилизации интерфейса через utility-классы.

CORS (Cross-Origin Resource Sharing) - механизм настройки разрешенных источников, с которых frontend может обращаться к backend.

CSRF (Cross-Site Request Forgery) - класс атак, при которых злоумышленник пытается выполнить действие от имени пользователя через его активную сессию.

XSS (Cross-Site Scripting) - класс атак, при которых вредоносный скрипт внедряется в страницу или данные.

HTTPS (HyperText Transfer Protocol Secure) - защищенный протокол передачи данных между браузером и сервером.

WAF (Web Application Firewall) - веб-фаервол для фильтрации и блокировки вредоносных HTTP-запросов.

Docker - технология контейнеризации приложений.

Docker Compose - инструмент запуска нескольких связанных контейнеров.

Telegram webhook - endpoint backend, на который Telegram отправляет события бота.

Tracking code / трек-код - уникальный код заявки, по которому пользователь и администратор могут идентифицировать запись.

КНБ РК - Комитет национальной безопасности Республики Казахстан.

РК - Республика Казахстан.

ИИН - индивидуальный идентификационный номер.

ФИО - фамилия, имя, отчество.

## 2. Общие сведения о системе

Система представляет собой двуязычный веб-портал Комитета национальной безопасности Республики Казахстан. Портал предоставляет гражданам и кандидатам публичную информацию о Комитете, направлениях деятельности, поступлении на службу и учебу, нормативной базе, контактах территориальных подразделений, а также сервисы регистрации, входа, личного кабинета, подачи первичной заявки на поступление и прохождения психологического тестирования.

Система состоит из двух основных частей:

- frontend-приложение на Next.js, React, TypeScript и Tailwind CSS;
- backend-приложение на FastAPI, SQLAlchemy, Pydantic и Alembic.

Портал поддерживает две локали:

- русский язык: `/ru`;
- казахский язык: `/kk`.

Корневой маршрут `/` перенаправляет пользователя на русскую версию `/ru`.

## 3. Роли пользователей

### 3.1. Неавторизованный пользователь

Неавторизованный пользователь может просматривать публичные разделы портала, пользоваться навигацией, переходить между языковыми версиями, открывать нормативные ссылки, просматривать контакты, подавать первичную заявку на поступление и переходить на страницы входа или регистрации.

Неавторизованный пользователь не может открыть личный кабинет, пройти основной полноэкранный психологический тест `primary-selection`, просмотреть свои результаты тестирования и открыть административную панель.

### 3.2. Кандидат

Кандидат - пользователь, зарегистрированный через email и пароль либо созданный после Telegram-входа. Роль кандидата используется для доступа к личному кабинету и прохождения сервисов портала.

Кандидат может:

- входить в личный кабинет;
- просматривать данные своего профиля;
- видеть кандидатскую анкету и ее статус, если она создана;
- переходить к разделам поступления на службу, поступления на учебу и психотестирования;
- проходить доступное психологическое тестирование;
- просматривать сохраненные результаты психологического тестирования;
- выходить из учетной записи.

### 3.3. Модератор

Модератор - служебная роль для пользователей административного контура. В текущей реализации административная панель защищена отдельной admin-session логикой. Доступ к административным API предоставляется только при наличии специального JWT-признака `admin_session`.

Модераторская логика включает:

- просмотр обращений и первичных заявок;
- изменение статусов обращений;
- изменение статусов кандидатских заявок;
- добавление комментария модератора к кандидатской заявке;
- просмотр результатов психологического тестирования;
- инициирование аудиторских записей при изменении статусов.

### 3.4. Администратор

Администратор - пользователь административного контура с полным доступом к панели управления. В текущей реализации вход в административную панель выполняется в две ступени:

1. пользователь должен быть обычным авторизованным пользователем с email, указанным в настройке `ADMIN_PORTAL_ALLOWED_USER_EMAIL`;
2. после этого пользователь должен пройти отдельный вход в административную панель через `ADMIN_PANEL_EMAIL` и `ADMIN_PANEL_PASSWORD_HASH`.

Администратор может работать с теми же разделами панели, что и модератор: dashboard, обращения, кандидаты, результаты тестирований.

## 4. Общая структура публичного интерфейса

### 4.1. Шапка сайта

Шапка сайта отображается на локализованных публичных страницах и содержит:

- эмблему КНБ;
- короткое название бренда;
- основное меню;
- переключатель режима доступности;
- переключатель языка;
- кнопку входа или кнопку перехода в личный кабинет;
- мобильное меню.

Основные пункты навигации:

- Главная;
- О Комитете;
- Направления деятельности;
- Поступление на службу;
- Поступление на учебу;
- Психотестирование;
- Нормативная база;
- Контакты.

Кнопка с эмблемой и коротким названием ведет на главную страницу текущей локали.

Кнопка режима доступности включает или выключает визуально усиленный режим. При включении значение сохраняется в `localStorage`, а на `body` добавляется CSS-класс `visually-enhanced`. Режим увеличивает базовый размер шрифта и усиливает контраст.

Переключатели `kk` и `ru` переводят пользователя на главную страницу соответствующей языковой версии.

Если пользователь не авторизован, в шапке отображается кнопка `Вход`. Нажатие ведет на `/{locale}/login`.

Если пользователь авторизован, вместо кнопки входа отображается круглая кнопка с иконкой пользователя. Нажатие ведет в личный кабинет `/{locale}/account`.

На мобильных экранах навигация открывается через кнопку меню. В мобильном меню доступны все основные разделы, а также вход или переход в личный кабинет.

### 4.2. Подвал сайта

Подвал сайта содержит:

- эмблему и короткое название КНБ;
- краткий информационный текст;
- навигационные колонки;
- ссылки на основные публичные разделы;
- блок контактов;
- город и страну;
- телефон `1400`;
- строку copyright.

Ссылки в подвале ведут на локализованные страницы текущего языка.

## 10. Раздел «Подача заявки на поступление»

Маршрут: `/{locale}/appeals`.

Раздел предназначен для подачи первичной заявки на поступление на службу или учебу.

### 10.1. Назначение формы

Форма собирает контактные данные пользователя, выбранный вариант поступления, интересующее направление и краткое описание цели обращения. После отправки система регистрирует заявку, выдает трек-код и показывает инструкцию по отправке резюме.

### 10.2. Визуальные шаги

Слева от формы отображается пошаговая инструкция:

1. Выберите службу или учебу.
2. Заполните все поля.
3. Нажмите кнопку отправки.
4. Отправьте необходимые документы на адрес почты, который появится после отправки.
5. Ожидайте обратной связи.

### 10.3. Поля формы

Форма содержит поля:

- `Вариант поступления` - обязательный выпадающий список;
- `ФИО` - обязательное поле, от 3 до 255 символов;

## 13. Регистрация и вход

### 13.1. Страница входа

Маршрут: `/{locale}/login`.

Страница входа содержит:

- информационный блок с эмблемой и описанием;
- переключатель способа входа;
- форму Telegram-входа;
- форму входа по email и паролю;
- блок защищенной авторизации;
- ссылку на регистрацию.

Доступные способы входа:

- Telegram;
- Email.

### 13.2. Telegram-вход

Telegram-вход состоит из следующих шагов:

1. Пользователь нажимает `Начать вход через Telegram`.
2. Frontend вызывает `/auth/telegram/start`.
3. Backend создает login challenge, возвращает `challenge_id`, `nonce`, `deep_link` и срок действия.
4. Браузер открывает Telegram deep link.
5. Пользователь открывает бота и нажимает `Поделиться номером телефона`.
6. Telegram отправляет update на webhook backend.
7. Backend проверяет, что `contact.user_id` совпадает с `message.from.id`.
8. Backend переводит challenge в статус `verified`.
9. Frontend периодически проверяет `/auth/telegram/status/{challenge_id}`.
10. При статусе `verified` frontend вызывает `/auth/telegram/complete`.
11. Backend создает или обновляет пользователя, выдает access token и refresh cookie.
12. Пользователь перенаправляется в личный кабинет.

Кнопка `Начать вход через Telegram` создает заявку на вход и открывает Telegram.

Кнопка `Открыть Telegram` становится активной после создания challenge и повторно открывает deep link.

Если срок challenge истек, пользователь видит сообщение о необходимости начать вход заново.

### 13.3. Вход по email и паролю

Форма входа содержит:

- email;
- пароль;
- кнопку `Войти`.

После успешного входа пользователь перенаправляется в личный кабинет.

При ошибке отображается сообщение backend или общее сообщение о невозможности входа.

### 13.4. Страница регистрации кандидата

Маршрут: `/{locale}/register`.

Регистрация кандидата выполняется через email и пароль.

Форма регистрации содержит:

- имя;
- фамилию;
- дату рождения;
- номер телефона;
- email;
- пароль;
- кнопку `Создать аккаунт`.

После успешной регистрации backend создает:

- пользователя с ролью `candidate`;
- кандидатскую заявку с трек-кодом `CAN...`;
- refresh-сессию;
- access token.

После регистрации пользователь перенаправляется в личный кабинет.

### 13.5. Требование к паролю

Пароль должен содержать:

- не менее 10 символов;
- заглавную букву;
- строчную букву;
- цифру.

Если пароль не соответствует политике, frontend показывает локализованное сообщение об ошибке.

### 13.6. Сессия пользователя

Access token хранится в `sessionStorage` браузера.

Refresh token хранится в HttpOnly cookie `knb_refresh_token`.

Если access token отсутствует или истек, frontend вызывает `/auth/refresh`. Запросы обновления сессии дедуплицируются через общую promise-переменную, чтобы избежать конфликтов при ротации refresh token.

При logout frontend вызывает `/auth/logout`, backend отзывает refresh session и удаляет cookie.

## 15. Психологическое тестирование

### 15.1. Страница списка тестов

Маршрут: `/{locale}/psychological-testing`.

Страница описывает первичный психологический тест и структуру тестирования.

Кнопки:

- `Начать тестирование` - ведет на `/{locale}/psychological-testing/primary-selection`;
- `Вернуться к поступлению` - ведет на `/{locale}/careers/admission`.

Страница отображает секции:

- числовые задания - доступно сейчас;
- наглядно-образные задания - доступно сейчас;
- задания на память - будет добавлено;
- итоговая интерпретация - после ключей оценки.

### 15.2. Полноэкранный тест primary-selection

Маршрут: `/{locale}/psychological-testing/primary-selection`.

Для этого теста скрываются обычные шапка и подвал сайта. Тест работает в полноэкранном режиме.

Перед показом вопросов frontend проверяет авторизацию через `/auth/me`.

Если пользователь не авторизован, отображается экран:

- заголовок о необходимости входа;
- пояснение;
- кнопка `Войти`.

Кнопка `Войти` ведет на страницу входа текущей локали.

### 15.3. Секции теста

В текущей реализации доступны две готовые секции:

- numeric - 50 числовых вопросов;
- visual - 50 визуальных вопросов.

Секции memory и interpretation присутствуют как планируемые, но в основной runner попадают только секции со статусом `ready`.

### 15.4. Инструкции перед секцией

Перед каждой секцией отображается экран инструкций.

На экране инструкций показываются:

- название секции;
- количество вопросов;
- текст инструкций;
- текущий номер секции;
- кнопка `Начать раздел`.

Таймер на экране инструкций не уменьшается. Отсчет начинается только после перехода к вопросам.

### 15.5. Таймер

Общий таймер теста составляет 60 минут. Таймер уменьшается только в режиме вопросов. Когда время заканчивается, тест автоматически переходит в режим завершения.

### 15.6. Страницы вопросов

Вопросы отображаются страницами по 10.

На странице вопросов отображаются:

- номер секции;
- диапазон вопросов;
- оставшееся время;
- кнопка `Завершить тестирование`;
- список вопросов текущей страницы;
- поле или варианты ответа;
- кнопки перехода.

Кнопка `Предыдущие 10 вопросов` возвращает пользователя на предыдущую страницу текущей секции.

Кнопка `Следующие 10 вопросов` переходит к следующей странице текущей секции.

Кнопка `Следующий раздел` переводит пользователя к инструкциям следующей секции.

Кнопка `Завершить тестирование` завершает тест досрочно.

Переход дальше заблокирован, пока пользователь не ответит на все вопросы текущей страницы.

### 15.7. Завершение и сохранение результата

После завершения теста отображается экран:

- `Тестирование завершено`;
- текст о сохранении ответов;
- статус сохранения;
- количество отвеченных вопросов;
- номер секции.

Frontend отправляет результат в `/psychological-tests/results`.

Сохраняются:

- slug теста;
- название теста;
- общее количество вопросов;
- количество отвеченных вопросов;
- длительность теста;
- оставшееся время;
- результаты по секциям;
- ответы пользователя.

Если сохранение успешно, отображается `Результат сохранен`.

Если сохранить не удалось, отображается ошибка сохранения.

### 15.8. Остальные тренировочные тесты

Для тестов attention, memory, logic и stress-resilience используется базовый runner. Он показывает вопросы, варианты ответов, кнопку `Завершить тест` и после завершения может показать справочный процент результата для тестов с оценочными вариантами.

## 16. Административная панель

### 16.1. Точка входа

Маршрут: `/admin`.

Административная панель не использует обычную публичную шапку и подвал.

При открытии страницы выполняется проверка:

1. пользователь должен быть авторизован в обычной пользовательской сессии;
2. email пользователя должен совпадать с `NEXT_PUBLIC_ADMIN_PORTAL_ALLOWED_USER_EMAIL`;
3. если admin session уже есть в `sessionStorage`, открывается панель;
4. если admin session отсутствует, показывается форма входа в админ-панель;
5. если проверка не пройдена, отображается 404.

### 16.2. Форма входа в админ-панель

Форма содержит:

- email администратора;
- пароль администратора;
- кнопку `Войти`;
- блок ошибки.

Кнопка `Войти` вызывает `/auth/admin/login`. Backend проверяет:

- обычную авторизацию пользователя;
- разрешенный email пользователя портала;
- email административной панели;
- хэш пароля административной панели.

При успехе backend выдает отдельный access token с признаком `admin_session: true`. Frontend сохраняет его в `sessionStorage` под ключом `knb-admin-access-token`.

### 16.3. Общая структура панели

- заголовок `Панель управления`;
- кнопку `Обновить`;

- рабочую область выбранной вкладки.

Верхние карточки показывают:

Дополнительные карточки показывают:

- количество одобренных заявок;

### 16.5. Вкладки панели

Доступны вкладки:

- Dashboard;
- Кандидаты;

Нажатие на вкладку меняет активный рабочий раздел без перезагрузки страницы.

### 16.6. Dashboard

Вкладка Dashboard показывает:

- список кандидатских заявок, требующих рассмотрения;
- информацию об операторе текущей admin session.

### 16.7. Обращения

Вкладка `Обращения` отображает список заявок/обращений слева и детальную карточку выбранной записи справа.

В списке отображаются:

- тема;
- трек-код;
- статус.

При выборе обращения справа отображаются:

- тема;
- дата создания;
- текст сообщения;
- ФИО;
- email;
- телефон;
- ИИН, если указан;

- `received` - получено;
- `in_review` - на рассмотрении;
- `answered` - ответ подготовлен;
- `rejected` - отклонено.

Изменение статуса сразу отправляет PATCH-запрос в backend. Backend обновляет запись и создает audit log.

Вкладка `Кандидаты` содержит:

- строку поиска;
- фильтр по статусу;
- список кандидатских заявок;

Поиск работает по:

- имени;
- фамилии;
- отчеству;
- телефону;
- email;
- трек-коду.

Фильтр статуса содержит:

- все статусы;
- submitted;
- in_review;
- approved;
- rejected.

В списке отображаются ФИО, трек-код и статус.

В детальной карточке отображаются:

- ФИО кандидата;
- дата создания;
- email/телефон/Telegram username пользователя;
- телефон кандидата;
- ИИН, если указан;
- регион, если указан;
- уровень образования, если указан;
- желаемое направление, если указано;
- комментарий модератора;
- выпадающий список статуса.

Доступные статусы кандидатской заявки:

- `draft` - черновик;
- `submitted` - подано;
- `in_review` - на рассмотрении;
- `approved` - одобрено;
- `rejected` - отклонено.

Изменение статуса отправляет PATCH-запрос в backend вместе с комментарием модератора. Backend сохраняет статус, комментарий и audit log.

### 16.9. Тестирования

Вкладка `Тестирования` показывает результаты психологических тестирований.

Для каждого результата отображаются:

- `GET /api/v1/admin/psychological-tests/results` - все результаты для admin session.

### 17.4. Admin endpoint’ы

Endpoint’ы административной панели:

- `GET /api/v1/admin/dashboard`;
- `GET /api/v1/admin/appeals`;
- `GET /api/v1/admin/appeals/{appeal_id}`;
- `PATCH /api/v1/admin/appeals/{appeal_id}/status`;
- `GET /api/v1/admin/candidates`;
- `GET /api/v1/admin/candidates/{application_id}`;
- `PATCH /api/v1/admin/candidates/{application_id}/status`.

Все admin endpoint’ы требуют JWT с claim `admin_session: true`.

### 17.5. Основные модели данных

Система использует следующие основные модели:

- `User` - пользователь портала;
- `TelegramLoginChallenge` - заявка на Telegram-вход;
- `News` - новость;
- `Page` - статическая контентная страница;
- `Appeal` - первичная заявка на поступление;
- `CandidateApplication` - кандидатская анкета;
- `PsychologicalTestResult` - результат психологического тестирования;
- `RefreshSession` - refresh-сессия;
- `LoginAttempt` - журнал попыток входа;
- `RegionOffice` - региональный офис;
- `AuditLog` - журнал административных действий.

### 17.6. Трек-коды

Для первичных заявок используется префикс `APL`.

Для кандидатских заявок используется префикс `CAN`.

Трек-код должен быть уникальным и индексируемым в базе данных.

## 18. Требования к безопасности

### 18.1. Аутентификация

Система должна поддерживать вход через Telegram с подтверждением номера телефона и вход по email/паролю.

Telegram-вход должен проверять принадлежность отправленного контакта текущему Telegram-аккаунту через сравнение `contact.user_id` и `message.from.id`.

Email/пароль должны проверяться через безопасное хэширование пароля.

### 18.2. Сессии

Access token должен иметь ограниченный срок действия.

Refresh token должен храниться в HttpOnly cookie и сохраняться на backend в виде хэша в таблице refresh sessions.

При refresh старая refresh session должна отзываться, а новая создаваться.

При logout активная refresh session должна отзываться, cookie должна удаляться.

### 18.3. Защита административной панели

Административная панель должна использовать отдельную admin session.

Обычная роль пользователя не должна сама по себе давать доступ к административным API.

Admin API должны проверять claim `admin_session: true` и разрешенный email пользователя портала.

### 18.4. Защита от подбора

Система должна журналировать успешные и неуспешные попытки входа.

После превышения допустимого количества неуспешных попыток за период backend должен возвращать ошибку `Too many login attempts`.

Для Telegram-входа должно ограничиваться количество challenge-заявок с одного IP-адреса.

### 18.5. CORS и allowed hosts

Backend должен принимать запросы только с разрешенных origins, заданных в `CORS_ORIGINS`.

Backend должен ограничивать допустимые hosts через `ALLOWED_HOSTS`.

### 18.6. Telegram webhook

В production webhook должен проверять секретный заголовок `x-telegram-bot-api-secret-token`.

Секрет webhook должен задаваться через переменные окружения.

### 18.7. Валидация данных

Все публичные и административные входные данные должны валидироваться Pydantic-схемами.

Ограничения длины должны применяться к ФИО, телефону, теме, сообщению, паролю, комментариям и другим пользовательским полям.

### 18.8. Аудит

Изменение статуса обращения и изменение статуса кандидатской заявки должны создавать запись в audit log с:

- идентификатором оператора;
- действием;
- сущностью;
- идентификатором сущности;
- IP-адресом.

## 19. Требования к локализации

Портал должен поддерживать одинаковую функциональность на русском и казахском языках.

Каждый публичный маршрут должен существовать в обеих локалях.

Тексты кнопок, заголовков, описаний, статусов, ошибок и пустых состояний должны быть локализованы.

Переключение языка в шапке должно переводить пользователя на главную страницу выбранной локали.

## 20. Требования к интерфейсу и состояниям

Интерфейс должен показывать понятные состояния:

- загрузка данных;
- пустой список;
- ошибка запроса;
- успешная отправка;
- заблокированная кнопка во время отправки;
- необходимость авторизации;
- отсутствие результатов тестирования.

Формы должны использовать обязательные поля, ограничения длины и понятные сообщения об ошибках.

Кнопки должны визуально отличать активное, неактивное, выбранное и hover-состояние.

Навигация должна быть доступна как на desktop, так и на мобильных экранах.

## 23. Статусы

### 23.1. Статусы первичной заявки на поступление

- `received` - заявка получена;
- `in_review` - заявка находится на рассмотрении;
- `answered` - ответ подготовлен;
- `rejected` - заявка отклонена.

### 23.2. Статусы кандидатской заявки

- `draft` - черновик;
- `submitted` - заявка подана;
- `in_review` - заявка на рассмотрении;
- `approved` - заявка одобрена;
- `rejected` - заявка отклонена.

### 23.3. Статусы Telegram challenge

- `pending` - challenge создан;
- `awaiting_contact` - ожидается отправка номера телефона;
- `verified` - номер подтвержден;
- `consumed` - challenge использован для входа;
- `expired` - срок challenge истек.

## 24. Основные пользовательские сценарии

### 24.1. Подача заявки на поступление

1. Пользователь открывает раздел поступления на службу или учебу.
2. Нажимает кнопку подачи заявки.
3. Система открывает страницу формы.
4. Пользователь выбирает вариант поступления.
5. Пользователь заполняет ФИО, email, телефон, направление и сообщение.
6. Пользователь нажимает `Отправить заявку`.
7. Backend создает запись и возвращает трек-код.
8. Frontend показывает трек-код, статус и инструкцию по отправке резюме.

### 24.2. Регистрация кандидата

1. Пользователь открывает `/register`.
2. Заполняет имя, фамилию, дату рождения, телефон, email и пароль.
3. Нажимает `Создать аккаунт`.
4. Backend проверяет пароль и уникальность учетной записи.
5. Backend создает пользователя и кандидатскую заявку.
6. Frontend сохраняет access token и переводит пользователя в кабинет.

### 24.3. Вход через Telegram

1. Пользователь открывает `/login`.
2. Выбирает Telegram.
3. Нажимает `Начать вход через Telegram`.
4. Открывает Telegram-бота.
5. Делится номером телефона.
6. Backend подтверждает номер.
7. Frontend завершает вход и открывает кабинет.

### 24.4. Прохождение психологического теста

1. Пользователь входит в портал.
2. Открывает раздел психотестирования.
3. Нажимает `Начать тестирование`.
4. Система проверяет авторизацию.
5. Пользователь читает инструкцию секции.
6. Пользователь отвечает на вопросы страницами по 10.
7. После завершения frontend сохраняет результат.
8. Результат отображается в личном кабинете и в административной панели.

### 24.5. Модерация заявки

1. Администратор проходит обычный вход.
2. Открывает `/admin`.
3. Проходит отдельный вход в админ-панель.
4. Открывает вкладку `Обращения` или `Кандидаты`.
5. Выбирает запись.
6. Меняет статус.
7. Backend сохраняет изменение и audit log.
